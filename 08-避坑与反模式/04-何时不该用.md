# 何时不该用 Claude Code

> 知道什么时候**不**用，比知道什么时候用更重要。

## 强烈不建议

### 你能一句话描述 diff 的活
改一个变量名、加一个 `console.log`、改一行字串——
**直接动手比解释给 AI 快**。

### 高安全 / 合规修改
- auth、密钥处理、加密、权限边界
- 合规相关代码（HIPAA、PCI、GDPR）
- 必须人工写 + 人工 review，AI 顶多当审稿人

### 你说不清需求的活
"做个像 X 那样的东西"——它会做出"像，但不是"。
先把需求说清楚再开 Claude，不然你只是在花钱踩坑。

### 改架构 / 大型重构（直接让它整体动）
- "把这个 monolith 拆成微服务" → 让 Claude 自动跑 → 多半烂尾
- 必须人主导：拆 plan → 一次一切片 → 人审
- 见 [[../03-工作流篇/03-代码评审与重构]]

### 跑生产 / 接 prod 数据
- 不要在生产服务器开 Claude Code
- 不要让它用 prod credential
- 不要让它直接 deploy

## 谨慎使用

### 探索陌生大型代码库
- 用，但要 plan 模式 + 给时间盒
- 否则 "Infinite exploration" 反模式

### 对 review 高度依赖的项目
- AI review 可以**辅助**，不能**替代**人工最终把关
- Armin Ronacher 的观点：人类 review 是真正瓶颈
- 来源：https://lucumr.pocoo.org/2026/2/13/the-final-bottleneck/

### 文档自动生成
- 自动文档"自信地错"比没文档更糟
- antirez 在 HN 提的 Deep Wiki 警告
- 来源：https://news.ycombinator.com/item?id=44049411

## 判断框架

拿到一个任务，问自己三个问题：

1. **我能在 30 秒内自己做完吗？** → 能，别开 Claude。
2. **出错了我能快速发现并回滚吗？** → 不能，先建 checkpoint 或 git commit，再开 Claude。
3. **这个任务需要我真正理解结果吗？** → 需要，全程参与，不要让它自动跑完再看。

三个都是"否/不能/不需要"的任务，才是最适合放手让 Claude 跑的。

## 一个容易忽视的场景：deadline 压力下

越急越容易把不该交给 Claude 的活交给它——"先让它跑着，我去干别的"。
结果：它跑偏了，你回来发现要重做，比自己写还慢。

deadline 压力下，Claude Code 最适合做的是**加速你已经想清楚的部分**，而不是替你想清楚。
