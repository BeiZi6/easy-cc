# 无限循环与失败重试

## 表现

- Edit 反复失败，它一直重试同一个 old_string
- 测试一直红，它一改再改还是错
- "再来一次"、"再试一次"刷屏

## 根因

- LLM 的修复策略偏增量：错了 → 改一点 → 没好 → 再改一点
- 没有"停下来重新理解"的本能
- 没有外部约束的话，能在错误轨道上跑很久

## 你该做的

- **同一个错连续 2 次没解决，立刻停下**
- 让它**先解释根因**，再动手
- 如果根因解释不清，**回到读代码**——不是改代码
- 必要时切换方法（不同算法、不同库、不同思路）
- **彻底失败 → `/clear` 重开**，把已有代码状态告诉它

## 反模式：把"修复"当"重写"

- 它说"让我重写整个文件" → 危险信号
- 大概率是它已经迷糊了，宁可重新规划，不要让它整体重写

## hooks 保险

PreToolUse hook 可以在它对同一文件第 3 次失败时拦截，强制提示：「停下，先解释为什么前两次都失败」。

详见 [[../04-配置与定制/04-Hooks钩子系统]]。

## 实战案例

### 案例 1：Edit 工具死循环
Claude 想改一个函数，但 `old_string` 匹配不上（文件被它自己改过了），于是反复重试同一个 Edit，每次都失败。
**处理**：直接打断，让它先 Read 文件确认当前内容，再重新构造 Edit。

### 案例 2：测试红了改不好
单元测试一直报同一个错，它每次改一点点，越改越乱，最后把原本好的逻辑也改坏了。
**处理**：`/clear`，把测试文件和报错贴给它，让它先解释根因，不许动代码，解释清楚了再改。

### 案例 3：subagent 嵌套循环
orchestrator 让 subagent 修 bug，subagent 失败后 orchestrator 又派了一个，循环往复。
**处理**：在 orchestrator 的 CLAUDE.md 里加规则：同一个 bug 超过 2 次 subagent 失败，停下来报告，不要继续派。

## 识别信号

这些词出现在 Claude 输出里，说明它可能已经在循环了：
- "让我再试一次"
- "我来重新尝试"
- "这次我换个方式"（连续出现 3 次以上）
- 同一个文件被 Edit 超过 5 次
