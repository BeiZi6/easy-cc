# 与 Claude 对话的方式

## 怎么提需求最好用

跟 Claude Code 对话，核心原则是：**说清楚你要什么，别教它怎么做。**

好的需求包含三要素：

1. **目标**——你想达到什么效果
2. **约束**——必须用什么库、不能动哪些文件、性能要求
3. **验收标准**——怎样算完成

```
❌ 烂需求：
"在 src/auth.ts 第 45 行加一个 if 判断，检查 token 是否过期"

✅ 好需求：
"登录 token 过期后用户没被踢出，应该自动跳转到登录页。
不要动 src/api/ 下的文件。改完跑一下现有测试确保没挂。"
```

给目标而不是步骤，**Claude 往往能找到比你预想更好的实现方式**。但如果你确实知道正确步骤（比如特定 API 的调用方式），直接告诉它也没问题。

## 三种粒度

根据任务复杂度，选择合适的描述粒度：

**一句话**——小修小补直接说：

```
修一下登录页跳转 bug，token 过期应该回到 /login
```

**半页描述**——中等任务附上复现步骤和期望行为：

```
问题：用户点击"导出 CSV"后页面卡死
复现：登录 → 进入报表页 → 选择日期范围超过 1 年 → 点导出
期望：大数据量时用流式下载，不要一次性加载到内存
```

**完整 spec**——大功能贴需求文档或截图，让它先列计划再动手：

```
我要加一个评论系统，需求文档在 @docs/comment-spec.md
先列个实现计划给我看，别直接写代码
```

## 附加上下文的方式

Claude Code 支持多种方式把额外信息塞进对话：

| 方式 | 用途 | 示例 |
|------|------|------|
| `@文件名` | 引用单个文件 | `看一下 @src/utils.ts 有没有内存泄漏` |
| `@文件夹/` | 引用整个目录 | `@src/components/ 这些组件的命名风格统一一下` |
| 拖图片 | 截图、报错、设计稿 | 直接拖进终端输入区 |
| `!命令` | 把命令输出塞进上下文 | `!cat package.json` |

> [!tip]
> `@` 引用文件比手动粘贴内容好得多——Claude 能看到文件路径，后续修改时知道改哪里。

## 多轮对话节奏

跟 Claude Code 协作的最佳节奏是：**改 → 看 → 反馈 → 再改**。

每轮聚焦一件事。别在一条消息里塞五个需求，那样它容易顾此失彼。

```
第 1 轮：「把用户列表改成分页加载」
第 2 轮：「分页 OK 了，但加载时没有 loading 状态，加一下」
第 3 轮：「loading 加了，再加个空状态的提示」
```

如果一次性说"加分页、加 loading、加空状态、加错误处理、加骨架屏"，它可能每个都做了但质量参差不齐。拆开来，每轮验收一个，最终质量更高。

## 打断与修正

发现它跑偏了？两种方式打断：

- **Esc**：立即中断当前工具调用，它会停下来等你指示
- **直接说"停，换个方向"**：比删掉消息重新编辑 prompt 更省 token，也更自然

> [!warning]
> 打断后不需要重复之前的上下文。Claude 记得整个对话历史，直接说新方向就行。

```
你：停，这个方向不对。不要用 WebSocket，改用 SSE 实现
Claude：好的，我换成 Server-Sent Events 来实现...
```
