# Plan 模式与 Auto 模式

Claude Code 有两个关键"开关"，决定它是「先报备再动手」还是「自己冲」。理解这两个模式，能让你在安全和效率之间找到最佳平衡。

## Plan 模式

### 是什么

Plan 模式下，Claude 禁止一切写操作。它只能用 Read、Grep、Glob 这些只读工具探索代码，然后输出一份"我打算这么做"的计划。

你审核计划、确认方向没问题后，它才会进入执行阶段。

### 进入 / 退出

```
进入：输入 /plan 或使用快捷键 Shift+Tab 切换
退出：批准计划后自动进入执行，或手动切回普通模式
```

### 什么时候用

- 改动涉及多文件、跨模块——先看看它打算怎么改
- 不确定它是否理解了你的需求——让它先复述方案
- 你想审一下技术方案再动手——避免返工

### 什么时候不用

- 改一行小 bug——直接让它改，Plan 反而浪费时间
- 你已经把方案讲得很清楚了——它不需要再"想一遍"

> [!tip]
> Plan 模式特别适合刚接手一个陌生项目时使用。让 Claude 先探索代码结构、列出它的理解，你来纠正偏差，比直接让它动手改效率高得多。

## Auto 模式

### 是什么

Auto 模式下，低风险的工具调用（读文件、编辑代码、跑测试等）自动放行，不再每次弹 `y/n` 确认。

### 进入方式

```
输入 /auto 切换到 Auto 模式
或启动时加参数进入
```

### 风险控制

Auto 不是"无脑全放行"。以下操作仍然会弹确认：

- 删除文件
- 修改生产环境配置
- git push / force push
- 任何在 deny 列表里的命令

### 怎么调

用 [[../04-配置与定制/03-权限系统permissions|permissions]] 精细化控制哪些命令自动放行、哪些必须确认：

```json
{
  "permissions": {
    "allow": [
      "npm test",
      "npm run build",
      "git diff",
      "git status"
    ],
    "deny": [
      "rm -rf",
      "git push --force"
    ]
  }
}
```

> [!warning]
> Auto 模式下 Claude 会连续执行多步操作。如果中间某步跑偏了，后面的步骤可能在错误基础上继续。建议搭配 hooks 给关键操作加一道闸——详见 [[../04-配置与定制/04-Hooks钩子系统]]。

## 推荐组合

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 探索陌生代码 | Plan | 先建立理解，避免乱改 |
| 改大功能 | Plan → Auto | 先审方案，确认后高效执行 |
| 跑测试循环 | Auto | 改→跑→改→跑，不想每次按 y |
| 重要重构 | 默认（每次确认） | 每步都看一眼，宁慢勿错 |
| CI / 无人值守 | headless + 严格 allowlist | 只允许预定义的安全操作 |

> [!tip]
> 最常用的组合是"Plan → Auto"：先用 Plan 让 Claude 列出方案，你确认后切到 Auto 让它一口气执行完。兼顾了安全和效率。
