# 上下文管理 — `/clear` 与 `/compact`

> 上下文是 Claude Code 的"短期记忆"。管不好，要么忘事，要么烧钱。

## 三个核心概念

在聊怎么管之前，先把三个词搞清楚：

**Context window** — 模型一次能"看到"的全部内容长度。你发的消息、Claude 的回复、读过的文件、工具调用结果，全部算在里面。目前 Claude 支持 200k 到 1M token 的窗口，但并不是越大越好（后面会讲为什么）。

**Compaction（压缩）** — 当上下文快撑满时，Claude Code 会自动把前面的对话压缩成摘要。原始细节丢了，但核心结论还在。你也可以手动触发。

**Reset（清空）** — 用 `/clear` 把整个对话历史一刀切掉，回到空白状态。干净利落，但之前的发现全没了。

## `/clear` — 完全清空

```
/clear
```

适合在你要切换到一个**完全无关的任务**时使用。比如刚帮你修完一个 bug，现在要写一个新功能的 spec — 之前的上下文只会添乱，不如清掉。

> [!warning] 副作用
> `/clear` 之后，Claude 对之前对话中发现的所有信息一无所知。如果有重要结论，先落盘再清 — 写进文件、写进 CLAUDE.md、或者让 Claude 先总结到一个 `notes.md` 里。

如果你不想完全丢失上下文但又想换任务，考虑用文件做持久化，参考 [[../03-工作流篇/05-长任务与记忆系统]]。

## `/compact` — 压缩

```
/compact
```

两种触发方式：

1. **自动触发**：上下文逼近窗口上限时，Claude Code 会自动执行 compaction。你会看到一条提示说上下文被压缩了。
2. **手动触发**：你觉得对话太长、开始跑偏、或者前面聊的东西已经不重要了，随时可以手动 `/compact`。

> [!tip] 压缩会丢什么
> 压缩后，文件的原始内容、早期对话的逐字细节大概率不在了。Claude 保留的是"摘要级别"的理解。如果后续还需要精确内容，它会重新读文件 — 所以问题不大，但你要知道这个机制。

手动 compact 还有个妙用：当你发现 Claude 开始"走神"（重复之前的错误、忘了你说过的约束），主动 compact 一下反而能让它重新聚焦，因为压缩后的摘要往往比冗长的原始对话更清晰。

## 实战节奏

不同规模的任务，上下文管理策略不一样：

**小任务（< 10 分钟）** — 从头到尾不需要 clear 也不需要 compact。一个对话搞定，结束后 clear 准备下一个。

**中任务（30 分钟 ~ 1 小时）** — 按阶段 compact。比如"读代码 → compact → 写实现 → compact → 写测试"。每次 compact 前，确保关键产物（代码、决策）已经落盘到文件里。

**大任务（跨小时甚至跨天）** — 拆成多个独立会话。用 `plan.md` 或 `progress.md` 文件串起来，每个会话开头让 Claude 读计划文件恢复上下文。这比在一个超长对话里硬撑要靠谱得多。

```markdown
<!-- progress.md 示例 -->
## 已完成
- [x] 数据库 schema 设计（见 specs/db-schema.md）
- [x] API 路由骨架

## 当前
- [ ] 实现用户注册接口

## 下一步
- [ ] 写集成测试
```

## 一个反直觉的事实

很多人觉得上下文越长越好 — "让它看到所有东西，它就能做对"。实际恰恰相反：

> [!warning] 上下文越长，模型越容易"走神"
> 研究和实践都表明，当上下文塞满了大量信息时，模型对关键指令的注意力会下降。短上下文 + 高质量提示，效果远好于长上下文里塞一堆"可能有用"的东西。

这就是为什么好的工作流强调"按需读取"而不是"一股脑全塞进去"。让 Claude 在需要时自己去读文件，比你提前把 20 个文件内容贴进对话要好。
