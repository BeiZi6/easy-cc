# 自动 PR 审查

> 让 Claude 当 24/7 reviewer，给每个 PR 写第一轮意见。

## 两种产品形态

### 1. GitHub Code Review（官方一键）
- 文档：https://code.claude.com/docs/en/code-review
- 开箱即用，开个开关就有"每 PR 自动 review"
- 适合：不想写 workflow 的团队

### 2. claude-code-action 自定义
- 仓库：https://github.com/anthropics/claude-code-action
- 解决方案合集：https://github.com/anthropics/claude-code-action/blob/main/docs/solutions.md
- 适合：想完全控制 review prompt / 限定路径 / 多团队规则不同

## 自定义 workflow 示例

```yaml
name: PR Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            请审查这个 PR 的 diff：
            1. 是否破坏向后兼容
            2. 是否有合理测试
            3. 错误处理是否到位
            4. 性能是否有隐患

            按严重度分级 [严重/中等/建议]，用 markdown 输出。
          allowed_tools: Read,Grep,Glob,Bash(git diff*),Bash(git log*)
```

## 实践建议

- **review prompt 不要太通用**——结合你的 codebase 写具体的检查项
- **按路径分流**：前端用一个 prompt，后端用另一个
- **不要让它自动改代码**——先让它评论，由人决定要不要让它动
- **跑两遍**：一遍 quick review（Sonnet），一遍 deep review（Opus 关键 PR）
- **看成本**：大 PR review 一次几毛钱，月度可能几百

## 反模式

- ❌ 让 Claude 在 PR 上来回硬怼（comment loop）
- ❌ 把 review 当 CI gate（false positive 会卡住所有人）
- ❌ 给它写权限——一旦它误改代码很难追
