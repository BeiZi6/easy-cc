# 调试与诊断循环

Bug 修复是开发中最吃时间的环节之一。Claude Code 可以大幅加速这个过程——但前提是你引导它做诊断，而不是让它瞎猜。

## 标准诊断五步

不管有没有 AI 辅助，靠谱的调试都遵循同一个循环：

### 1. 复现

让 bug 稳定复现，再动手。如果不能复现，你改的可能不是真正的问题。

```
这个接口偶尔返回 500。帮我写一个脚本，并发 50 次请求，看能不能稳定触发。
```

### 2. 缩小

砍到最小可复现样例。文件越少、步骤越短，定位越快。

```
这个测试文件有 200 行，帮我找出触发这个 assertion error 的最小输入。
逐步删减 fixture 数据，每次删完跑一遍测试，直到找到最小复现集。
```

### 3. 假设

列出 2-3 个候选原因，不要只盯着一个方向。

```
根据这个 stack trace 和相关代码，列出你认为最可能的 3 个原因，按可能性排序。
不要改代码，只分析。
```

### 4. 加探针

加日志、断点、print 来验证假设。**这一步不改业务逻辑**。

```
在这三个可疑位置加 console.log，打印变量的实际值。
跑一遍测试，把输出贴给我看。
```

### 5. 修复 + 加回归测试

确认了根因，再动手修。修完必须加测试，防止同一个 bug 再来。

```
根因确认了：并发时 cache 没加锁。修复它，然后加一个并发测试覆盖这个场景。
```

> [!tip] 这五步的关键是顺序
> 很多人（包括 Claude）的本能是跳过前四步直接到第五步。你的工作是把它拉回来。

## 反模式

### 凭直觉改一行看好没好

```
❌ "试试把这个 timeout 改成 5000？"
❌ "可能是这里少了个 await，加上试试？"
```

这不是调试，这是碰运气。就算碰对了，你也不知道为什么对了，下次还会踩。

### 同一个错误连改两次没解决

> [!warning] 两次失败 = 停下来
> 如果同一个报错改了两次还在，说明你的心智模型是错的。不要继续在同一个方向上加补丁。停下来，重新读代码，重新理解问题。

告诉 Claude：

```
这个方向试了两次都没用。停下来，重新分析 stack trace，
从头梳理数据流，找到真正的根因。
```

### 把异常捕获掉就行

```javascript
// ❌ 这不是修 bug，这是藏 bug
try {
  riskyOperation();
} catch (e) {
  // ignore
}
```

如果 Claude 给你这种"修复"，直接打回去。异常存在是有原因的，吞掉它只会让问题在更远的地方爆炸。

## 让 Claude 做诊断而不是猜

prompt 的质量直接决定 Claude 是在做诊断还是在瞎改。

**好 prompt：**

```
这个测试报错 "TypeError: Cannot read property 'id' of undefined"。
stack trace 在下面。

先只读相关代码 + 加 print，不改任何业务逻辑。
把你的怀疑列出来，我们一起选一个验证。

[粘贴 stack trace]
```

**烂 prompt：**

```
修一下这个 bug。
```

区别在哪？好 prompt 做了三件事：
1. 给了具体的错误信息和上下文
2. 明确限制了行动范围（只读 + 加 print）
3. 要求先分析再行动

烂 prompt 等于说"随便改到不报错就行"，Claude 很可能给你一个 `if (!obj) return` 了事。

## 工具配合

### diagnose skill

如果你安装了 `diagnose` skill，可以直接用：

```
/diagnose 这个测试在 CI 上偶尔失败，本地跑不出来
```

它会自动走"复现→缩小→假设→探针→修复"的循环，比手动引导更省心。

### git bisect 找引入点

当你知道"以前是好的，现在坏了"，但不知道哪个 commit 引入的问题：

```
帮我用 git bisect 找出引入这个 regression 的 commit。
测试命令是 npm test -- --grep "user login"。
好的 commit 是上周五的 abc1234。
```

Claude 会自动驱动 bisect 流程，二分查找到问题 commit，然后分析那个 commit 的改动。

### 让 Claude 看 stack trace

直接把完整的 stack trace 粘贴给 Claude，比你自己描述问题有效得多：

```
生产环境报了这个错，帮我分析根因：

Error: ECONNREFUSED 127.0.0.1:5432
    at TCPConnectWrap.afterConnect [as oncomplete] (net.js:1141:16)
    at Protocol._enqueue (/app/node_modules/pg/lib/protocol.js:350:23)
    at Client.query (/app/src/db/client.ts:45:12)
    at UserRepository.findById (/app/src/repos/user.ts:23:18)
    ...
```

Claude 能从 stack trace 中读出调用链、定位到具体文件和行号，然后去读那些代码来分析问题。这比"数据库连接有时候报错"这种描述强太多了。

> [!tip] 给 Claude 的信息越具体，它的诊断越准
> 错误信息、stack trace、复现步骤、环境信息——这些都是诊断的燃料。别让它在信息不足的情况下猜。
