# 代码评审与重构

Claude Code 既能当 reviewer 帮你审代码，也能当重构执行者帮你改代码。但这两件事的风险等级完全不同——审代码最多漏看，重构搞砸了可能要 `git reset --hard`。

## 让 Claude 当 Reviewer

### /review 命令

最直接的用法，在有未合并 PR 的分支上执行：

```bash
/review
```

Claude 会读取当前分支与 main 的 diff，给出逐文件的评审意见。它会关注逻辑错误、边界条件、命名不一致等常见问题。

### /security-review 安全视角

专门从安全角度审视变更：

```bash
/security-review
```

它会重点检查：注入风险、认证/授权漏洞、敏感信息泄露、不安全的依赖使用等。适合在合并前跑一遍，尤其是涉及用户输入、API 接口、权限控制的改动。

### 自定义 review prompt

默认的 review 视角不一定适合你的项目。你可以在 CLAUDE.md 里写明你的 review 标准：

```markdown
## Code Review 标准

- 所有公开 API 必须有 JSDoc
- 禁止 any 类型逃逸
- 数据库查询必须走 repository 层，不能在 controller 里直接写 SQL
- 错误处理必须用自定义 AppError，不能裸抛 Error
```

这样 Claude 在 review 时会自动对照这些标准。

### 多角色 review

对于重要的 PR，可以让 Claude 从多个角度分别审视：

```
帮我从三个角度 review 这个 PR：
1. 性能：有没有 N+1 查询、不必要的循环、内存泄漏风险
2. 可维护性：命名是否清晰、抽象层次是否一致、新人能否看懂
3. 安全：输入校验、权限检查、敏感数据处理

每个角度单独给结论。
```

Claude 会用 subagent 分别处理每个角度，最后汇总结果。这比一次性让它"全面 review"效果好得多——聚焦的 prompt 产出聚焦的结果。

## 让 Claude 重构

### 危险信号

看到这些话从你嘴里说出来，请立刻警觉：

- "帮我重写整个文件"
- "整体优化一下结构"
- "把这个模块重构得更优雅"

> [!warning] 这类指令大概率失控
> Claude 会很乐意大刀阔斧地改，但改完你可能发现：测试全挂、行为变了、改动太大根本没法 review。模糊的重构指令 = 赌博。

### 安全姿势

重构的核心原则：**行为不变，结构变好**。要做到这一点：

1. **重构前先有测试**——没有测试覆盖的代码，重构就是在走钢丝。先让 Claude 补测试，确认行为，再动结构。

2. **一次只改一个轴**——要么只改命名，要么只拆函数，要么只提抽象。不要同时做两件事。

3. **每步跑测试**——改完一个小步骤就跑一次测试。告诉 Claude：

```
每做完一个重构步骤就跑 npm test，确认没有破坏行为再继续下一步。
```

### 重构清单

以下是适合让 Claude 做的、风险可控的重构类型：

| 重构类型 | 示例 prompt |
|---------|------------|
| 命名一致化 | "把所有 `getUserInfo` 统一改成 `getUser`，用 semantic rename" |
| 死代码清理 | "找出所有没被引用的导出函数，列出来让我确认后删除" |
| 函数拆小 | "这个函数 80 行太长了，把每个 if 分支提取成独立函数" |
| 重复消除 | "这三个文件里的校验逻辑几乎一样，提取成共享的 validate 函数" |
| 提取参数对象 | "这个函数有 6 个参数，提取成一个 Options 接口" |

> [!tip] 每个 prompt 都是具体的、可验证的
> "提取成共享函数"比"优化代码结构"好一百倍。Claude 需要明确的终态。

## "改架构"级别的重构

当你要做的不是"改个名字"而是"换掉整个数据层"或"从单体拆微服务"，那就不是一个 prompt 能搞定的事了。

正确的流程：

1. **先写 spec**——用 [[01-Spec驱动开发]] 的方式，把目标架构、迁移策略、风险点写清楚
2. **拆计划**——把大重构拆成多个独立可交付的垂直切片
3. **一次一个 PR**——每个切片是一个完整的、可独立部署的变更

```
错误做法：先把所有旧代码重构完，再补新功能
正确做法：每个 PR 包含"旧→新"的一个完整路径，旧代码逐步退役
```

> [!warning] 绝对不要让 Claude 一口气做架构级重构
> 它会很开心地改几十个文件，然后你花三天时间 review 一个无法理解的巨型 diff。垂直切片、逐步推进，是唯一可控的方式。
