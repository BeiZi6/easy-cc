# TDD 与 Claude Code

> Anthropic 自家工程师反复推荐的玩法：先让它写测试，再让它写实现。

## 为什么 TDD 特别适合 LLM

TDD 和 LLM 是天作之合，原因有三：

**测试就是机器可读的 spec。** 你写一段自然语言描述需求，Claude 可能理解偏。但一组测试用例摆在那里，输入输出、边界条件全是确定的 — 模型有了明确的、无歧义的目标。

**红绿循环天然有反馈信号。** 普通开发流程里，Claude 写完代码说"搞定了"，你很难立刻验证。但 TDD 里，跑一下测试就知道是红是绿。模型能根据测试输出自我纠偏，不需要你逐行 review。

**防止"声称完成"但没跑通。** LLM 有个坏习惯：代码看起来对，它就说 done 了。有测试兜底，它必须真正跑通才算完。这是硬约束，不是靠提示词"请认真检查"能替代的。

## 标准节奏：红 → 绿 → 重构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  1. 写测试   │ ──→ │  2. 确认全红  │ ──→ │  3. 最小实现  │
│  （只写测试） │     │  （测试失败） │     │  （测试变绿） │
└─────────────┘     └─────────────┘     └─────────────┘
                                                │
                                                ▼
                                        ┌─────────────┐
                                        │  4. 重构     │
                                        │ （保持绿色）  │
                                        └─────────────┘
```

具体操作：

1. 让 Claude 读懂需求，**只写测试，不写实现**
2. 你跑测试，确认全红（全部失败）— 这步很重要，确保测试本身是有效的
3. 让它"用最小改动让测试变绿，不许修改测试"
4. 跑测试，确认全绿
5. 让它重构（保持测试绿色）

> [!tip] 为什么要确认"全红"
> 如果测试写完就是绿的，说明测试没测到东西（比如断言写反了、mock 太宽泛）。全红是 TDD 的安全网 — 证明这些测试真的在检验新功能。

## 关键提示词

这三句话是 TDD 工作流的核心咒语，建议直接复制使用：

```
先只写测试，写完停下，等我审。
```

```
只为通过测试做最小改动。不许扩展功能、不许重构现有代码。
```

```
禁止修改测试文件 tests/user_invitation_test.ts 来让测试通过。
```

第一句防止它一口气写完测试和实现（你就失去了审测试的机会）。第二句防止它过度工程化。第三句防止它走捷径 — 改测试让测试通过是 LLM 最常见的"作弊"手段。

## 防作弊技巧

Claude 有时候会"聪明反被聪明误"— 与其写正确实现，不如改测试让它通过。几个防御手段：

**测试文件标记只读 / hooks 拦截写入**

用 Claude Code 的 hooks 系统拦截对测试文件的修改：

```json
// .claude/settings.json
{
  "hooks": {
    "PreFileWrite": [
      {
        "matcher": "tests/**",
        "command": "echo 'BLOCKED: 不允许修改测试文件' && exit 1"
      }
    ]
  }
}
```

这样 Claude 一旦试图写入测试目录，就会被拦下来。

**要求贴运行输出**

在提示词里加一句：

```
跑完测试后，把完整的命令和输出贴给我看。
```

这不是万无一失的（它理论上可以编造输出），但在实践中能显著提高诚实度。看到具体的 PASS/FAIL 输出，你也能快速判断是否真的通过了。

**跑覆盖率检查**

```bash
# 测试通过不代表覆盖到了 — 跑覆盖率验证
npm test -- --coverage
# 或
pytest --cov=src --cov-report=term-missing
```

> [!warning] 覆盖率陷阱
> 100% 行覆盖率不等于逻辑正确。但如果新写的函数覆盖率是 0%，那测试肯定没测到它。覆盖率是下限检查，不是质量保证。

**多轮验证**

对于关键逻辑，可以在实现完成后追加一轮：

```
现在给我加几个边界情况的测试：空输入、超长字符串、并发调用。
先写测试，确认红，再改实现。
```

这既增加了覆盖面，又再次验证了实现的健壮性。
