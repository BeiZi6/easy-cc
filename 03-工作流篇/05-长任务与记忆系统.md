# 长任务与记忆系统

> 任何一个 session 装不下的活，都是"长任务"。

## 长任务的两难

Claude Code 的上下文窗口是有限的。当任务复杂到一定程度，你会撞上一个两难：

- **一个 session 塞太多**——上下文逐渐膨胀，模型开始"走神"，前面说过的事情它忘了，输出质量肉眼可见地下降。到了 compact 阈值，早期的细节会被压缩丢失。

- **拆成多个 session**——每次新开 session 都要重新解释背景、目标、当前进度。这些解释本身就在消耗 token，而且你很难保证每次都解释得一样完整。

核心矛盾：模型的工作记忆是易失的，但复杂任务需要持久的上下文。

## 解法：把工作记忆落到文件

既然模型的记忆会丢，那就别依赖它的记忆。把关键信息写到文件里，每个 session 开头让它读文件恢复状态。

推荐的文件结构：

```
.claude/
├── plan.md        # 总体计划：目标、范围、里程碑
├── tasks.md       # 任务清单 + 状态（todo/doing/done）
├── progress.md    # 当前进度：已完成什么、卡在哪、下一步
├── findings.md    # 探索过程中发现的事实、约束、意外
└── decisions.md   # 关键决策及其原因（轻量 ADR）
```

每个文件的职责不同：

| 文件 | 谁写 | 什么时候读 | 内容特点 |
|------|------|-----------|---------|
| plan.md | 你 + Claude | 项目开始时 | 稳定，很少改 |
| tasks.md | Claude | 每个 session 开头 | 清单式，状态明确 |
| progress.md | Claude | 每个 session 开头 | 简短，只保留当前状态 |
| findings.md | Claude | 需要时查阅 | 累积式，只追加 |
| decisions.md | 你 + Claude | 遇到分歧时 | 记录 why，不只是 what |

> [!tip] 用 planning-with-files skill 可以自动化这套流程
> 执行 `/planning-with-files` 会自动创建这些文件并维护更新节奏。

## Claude 自带的记忆机制

除了手动维护文件，Claude Code 本身也有一套记忆系统：

### CLAUDE.md 分层

```
~/.claude/CLAUDE.md          # 用户级：所有项目都生效的偏好
项目根目录/CLAUDE.md          # 项目级：这个项目的规范和约定
src/modules/auth/CLAUDE.md   # 子目录级：局部覆盖，只在这个目录下生效
```

层级越深优先级越高。适合放：编码规范、项目约定、常用命令、技术栈说明。

### 自动 memory

Claude Code 会自动记住一些东西——你的纠正、偏好、项目事实。这些记忆按类型分类存储：

- **user**：你的个人偏好（"我喜欢用 tabs 不用 spaces"）
- **project**：项目事实（"这个项目用 pnpm 不用 npm"）
- **feedback**：你对它输出的纠正（"不要用 default export"）

### 快速写入记忆

在对话中用 `#` 前缀可以快速写入一条记忆：

```
# 这个项目的测试用 vitest 跑，不是 jest
```

Claude 会把这条信息存入记忆，后续 session 自动加载。

### /memory 命令

```bash
/memory
```

打开记忆编辑界面，可以查看、修改、删除已有的记忆条目。适合定期清理过时的记忆。

## 节奏：长任务的 session 管理

一个可靠的长任务工作节奏：

### Session 开头

```
读一下 .claude/progress.md 和 .claude/tasks.md，恢复上次的工作状态。
告诉我当前进度和下一步计划。
```

### Session 中途

当 Claude 发现了重要信息（比如某个 API 的隐藏限制、某个依赖的 bug）：

```
把这个发现写进 .claude/findings.md。
```

### Session 结束前

```
更新 .claude/progress.md：记录今天完成了什么、下一步要做什么、有没有卡点。
更新 .claude/tasks.md 里的任务状态。
```

### 下一个 Session

回到第一步，循环。

> [!tip] 养成习惯：session 结束前永远让 Claude 更新 progress
> 这 10 秒钟的投入，能帮下一个 session 省掉 5 分钟的重新解释。

## 反模式

### 把所有记忆堆进 CLAUDE.md

CLAUDE.md 的内容每次对话都会加载到上下文里。如果你把所有笔记、发现、历史决策全塞进去，每个 session 一开始就吃掉大量 token，留给实际工作的空间就少了。

正确做法：CLAUDE.md 只放**当前仍然有效的规则和约定**。历史信息、探索笔记放到 findings.md 或 decisions.md，需要时再读。

### 写日志型记忆

```markdown
<!-- ❌ 没人会读的日志 -->
## 2026-05-20
今天我帮用户重构了 auth 模块，把 JWT 验证逻辑提取到了 middleware 里。

## 2026-05-19
今天修了一个 race condition 的 bug...
```

这种流水账式的记忆对 Claude 没有用。它需要的是**当前状态**和**可执行的下一步**，不是历史日记。

正确的 progress.md 长这样：

```markdown
## 当前状态
auth 模块重构完成，JWT 中间件已提取，测试通过。

## 下一步
- [ ] 把 session 管理从 Redis 迁移到 PostgreSQL
- [ ] 更新 API 文档

## 卡点
Redis session 的 TTL 逻辑和 PG 的 expires_at 语义不完全对等，需要确认边界情况。
```

简短、有状态、可行动。这才是有效的工作记忆。
